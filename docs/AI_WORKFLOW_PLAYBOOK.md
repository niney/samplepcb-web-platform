# AI_WORKFLOW_PLAYBOOK — 작업 방식 선택 기준

> 메인 세션(Fable)이 작업마다 진행 방식(직접 / 위임 / 병렬)을 **자율 결정**하기 위한 실증 기반 판단 기준.
> 갱신 주체: 메인 세션. 새 실증(성공·사고)이 나오면 하단 로그에 추가한다.

## 원칙 (사용자 확정 2026-07-04)

1. **퀄리티 > 비용.** 비용 절감을 이유로 품질 게이트(전수 감사·독립 실측·육안 1회)를 생략하지 않는다.
2. **방식은 Fable이 정한다 + 사용자는 원-체크.** 매번 선택지를 묻지 않는다. 결정과 근거(1~2줄)를 밝히는 지점이 체크 포인트다:
   - 플랜 모드 작업 → 계획서에 **"진행 방식: X — 근거"** 절을 명시. 플랜 승인 = 체크.
   - 플랜 없는 작업 → 착수 메시지에 방식+근거 1줄 고지 후 진행(사용자는 언제든 중단·변경 가능).
   - 사전 질문(AskUserQuestion)은 예외적으로만: 트레이드오프가 실제로 팽팽하거나(비용 대폭 증가, 파괴적 검증 필요), 다중 agent 병렬·worktree 같은 이례 구조일 때. 물을 때도 **추천은 단수**(하나를 정해 근거 제시) — 중립적 메뉴 나열 금지.
3. **이 문서는 기본값이지 족쇄가 아니다.** 상황이 다르면 이탈하되, 이탈 근거를 한 줄 고지한다.

## 방식별 특성 (실증 기반)

| 방식 | 적합 | 강점 | 약점 |
|---|---|---|---|
| **직접 (Fable 본체)** | 설계·구현이 얽혀 실시간 판단이 필요한 것 | 지시서 손실 0, 구현 중 설계 오류 즉시 수정 | 메인 컨텍스트 소모, 비용 최대 |
| **Opus 4.8 agent + Fable 전수 감사** | 설계가 계획으로 동결된 구현형(관례 미러링) | 메인 컨텍스트 보존(최대 실익), 구현자≠감사자 게이트, 비용↓ | 지시서 종속(설계 결함 복제), 왕복 마찰, 감사 비용 잔존, 실브라우저 사각 |
| **Fable agent (fork/general)** | Fable급 추론 + 컨텍스트 격리·병렬이 목적일 때 | 품질 동급, 격리 | 비용 절감 없음. fork=대화 상속·모델 고정, general=지시서 필요 |
| **Explore agent (병렬)** | 조사·탐색(파일 다수 훑기) | 메인 컨텍스트 보호, 병렬 | 리뷰·감사엔 부적합(발췌 읽기) |
| **다중 agent / Workflow** | 독립 파일 대량 마이그레이션·전수 점검, 관점 분리 병렬 리뷰 | 벽시계 단축, 커버리지 | 공유 파일 충돌(worktree 필요), Workflow 는 사용자 명시 opt-in 필요 |

## 작업 유형 → 기본값

- **신규 기능(설계 확정, 기존 관례 미러링)** → Opus 위임 + Fable 전수 감사. 실증 3회 모두 성공(하단 로그).
- **원인불명 디버깅·탐구형** → 직접. 가설-검증 루프는 지시서로 동결 불가.
- **시각·인쇄·픽셀 마감** → 직접 + 사용자 육안. agent 는 실브라우저 판단 불가(사례: 인쇄 드로어 겹침).
- **소규모 수정(지시서 작성 비용 > 구현 비용)** → 직접.
- **조사·사전 탐색** → Explore agent 병렬(독립 주제면 1메시지에 2~3개).
- **대량 독립 파일 작업·관점 분리 검증** → 다중 agent(충돌 시 worktree) 또는 Workflow(사용자 opt-in).
- **공유 DB 를 픽스처로 쓰는 검증** → 반드시 단일 agent(카운트 단언이 경합에 깨짐).
- **순차 의존 체인(계약→서버→화면)** → 분할 무익. 공유 파일(api-contract·i18n·문서) 충돌까지 고려하면 단일 구현자가 정답.

## 위임(Opus) 필수 절차 — 품질 게이트

1. **지시서 구성**: 승인된 계획서 경로 + ① 불변식(공유 DB·코어 비수정·g5 한정 예외·민감정보·i18n 동형·커밋 금지) ② 확정 설계 ③ 검증 절차(스냅샷→픽스처→실측→**정확 원복**) ④ 알려진 함정 ⑤ 보고 형식(변경 파일/이탈/검증 결과/원복 내역/남긴 이슈). 이 플레이북과 AGENTS.md 를 먼저 읽게 한다.
2. **완료 후**: Fable 이 diff **전수** 감사(요약 신뢰 금지) + 독립 실측(DB 상태·API 프로브·정적 검사 재실행) → UI 작업이면 사용자 육안 1회.
3. **이탈은 보고 의무** — 감사에서 타당성 판정(실증상 agent 의 자율 이탈이 계획보다 나은 경우가 있었음: affectedRows 멱등 판단, 예외 번호 충돌 조정).
4. 완료 보고 없이 idle 이면 재촉 1회(간헐 발생).

## 알려진 함정·제약

- **지시서 허점은 그대로 구현된다** — validUntil 결함은 지시서 결함이었고 감사에서야 회수.
- **실브라우저 사각** — 인쇄 미리보기 드로어 겹침은 DOM 검증으로 못 잡고 사용자 스크린샷으로 발견. UI 는 육안 1회를 절차에 포함.
- **Agent 호출에 effort 개별 지정 불가**(세션 effort 상속) · fork 는 모델 변경 불가(Opus 위임엔 general-purpose + model 지정).
- 머릿속 맥락은 전달되지 않는다 — 문서화 안 된 결정·취향은 지시서에 명시하거나 전달을 포기한 것으로 간주.
- 감사 비용은 사라지지 않는다 — 결함 밀도가 높은 작업이면 위임 이득이 역전될 수 있다(그런 유형은 애초에 직접).
